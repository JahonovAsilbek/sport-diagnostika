The user called out a tendency to jump from one task to another without
finishing the first ("sen bola bir ishni oxiriga yetkazmasdan boshqasiga o'tib
ketadigan odating bor ekan"). They prefer to **nail things down in writing**
before moving on (e.g. lock the big blocks into a roadmap doc, THEN break into
granular tasks — not both at once).

**Why:** keeps a large, not-yet-fully-visible project under control; avoids
half-done threads and noise.

**How to apply:** Work one unit at a time to completion. Before advancing a
phase, write the agreed decisions into the relevant `docs/*.md` and get
confirmation. Don't start the next sub-step (or spawn parallel work) until the
current one is closed out. Related: [[feedback_design_before_code]].

Project track prefixes for tasks: **BCKND** (backend), **DVPS** (devops),
**FRNTND** (frontend). Blocks live in `docs/ROADMAP.md`; granular tasks will go
in `docs/TASK.md`. See [[project_architecture]].
